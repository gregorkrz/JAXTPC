## High-level repo description
This is a differentiable LArTPC simulation repo written in JAX. We experiment with simulation parameter calibration using src/scripts/run_optimization.py. The loss landscape is generated using scripts/jobs_Landscape.sh.
Simulation calibration jobs are submitted to slurm using src/jobs/submit_jobs.py.

## Important insights

Add important insights about the repo that you learn here.

**"Conservative" E-field has two unrelated knobs — don't confuse them.** (1) GT/target distortion map: pass ``--electric-dist-path`` pointing at the curl-free npz generated by ``tools/efield_distortions.py --model conservative`` (vs default ``--model toy``). In ``submit_jobs_E_field_calibration.py`` this is ``REMOTE_CONSERVATIVE_EFIELD_NPZ`` (``sce_maps_jaxtpc_conservative_41.npz``), wired up **only** in ``profile_conservative_curl_sweep``; ``_submit_efield_tracks_sweep`` (used by both ``1k_tracks_sweep`` and ``100_tracks_sweep``) always defaults to the plain ``REMOTE_EFIELD_NPZ`` (``sce_maps_jaxtpc_41.npz``, non-conservative toy field) — confirmed via wandb (entity ``fcc_ml``, project ``jaxtpc-optimization``) that the latest ``100_tracks_sweep`` run (2026-07-03T12:35:49Z) used the plain map. (2) Learned SIREN-model conservativeness: ``--efield-mode`` (``parsing.py``) is now **deprecated/ignored** — the Efield model always uses the SIREN Δ-field representation (``tools/sce_siren.py``); the actual lever for penalizing a non-physical (non-curl-free) learned field is ``--penalize-rotor W`` (curl regularizer weight, evaluated on a ~200-point interior grid). That 100-track run did pass ``--penalize-rotor 1``.

Track ensemble: ``tools/random_boundary_tracks.py`` — ``generate_random_boundary_tracks`` builds N random x-face muons plus **three** fixed 1000 MeV chords that span East+West: ``Muon_diagCross_1000MeV``, ``Muon_throughEw_skew02_1000MeV``, ``Muon_throughWe_skew03_1000MeV`` (see that file for start/toward mm). Random-track energies are now balanced as evenly as possible across ``(1000, 500, 100)``; default ``N=12`` gives exactly ``4`` tracks at each energy, so total default tracks remain ``15`` with the 3 fixed chords. ``launch_2d_landscape_pairs.py`` uses the same list and passes ``--start-position-mm`` into ``2d_loss_landscape.py``. In ``plot_mixed_tracks_edep_wireplanes.py``, out-of-volume steps are dropped before ``build_deposit_data``; Plotly ``Scatter3d`` uses ``marker.line.width=0`` to reduce white halos.

E-field distortions (SCE): ``tools/efield_distortions.py`` is the **non-differentiable** generator (analytic toy field + SciPy-Euler drift integration); its ``__main__`` writes a per-side ``.npz`` (``east_*``/``west_*`` keys). ``tools/nonlocal_efield.py`` is the **differentiable** MLP counterpart — ``mode='potential'`` learns a scalar potential and returns ``E = E_bg − ∇φ`` (conservative by construction); also ``efield``/``correction`` modes. Helpers: ``sce_outputs`` (→ ``SCEOutputs``-style ``efield_correction``/``drift_corr_cm``), ``nominal_start_params`` (random hidden + **zeroed output layer** → starts at nominal field yet has nonzero gradient; an all-zero MLP is dead — zero gradient everywhere), ``flatten_params``/``zero_params``. The MLP field lives in ``SimParams.sce_models`` (previously a declared-but-unused field). ``tools/simulation.py`` ``DetectorSimulator(..., efield_model=FieldConfig)`` builds an MLP SCE factory that reads weights from ``sim_params.sce_models``; ``sce_factory`` now takes ``sim_params`` (4 call sites). The MLP operates in volume-**local** frame (one MLP serves both volumes; nominal ``Ex_local=+E0``).

Optimizable Efield in ``run_optimization.py``: pass ``Efield`` in ``--params`` (combinable with scalars, e.g. ``--params velocity_cm_us,Efield``) plus ``--electric-dist-path <npz>`` (the GT distortion the MLP learns to match). GT sim runs WITH the static ``.npz`` map (``load_sce_per_volume`` now also reads ``.npz``, not just ``.h5``); the diff sim learns the MLP. Options: ``--efield-mode {potential,efield,correction}``, ``--efield-hidden``, ``--efield-lr-mult``. The optimizer vector is ``[scalar block (log-normalized) | flat MLP weights (raw)]``; only the scalar coords are recorded in the param/grad trajectories (full ``final_p`` saved + ``result['efield']`` metadata). Not compatible with ``--optimizer newton`` or ``--init-from-wandb-run``. Per-step cost is dominated by the deposit-level forward+backward, NOT the MLP (8.6k weights ≈ free).

``run_optimization.py`` was refactored: helper groups now live in the top-level package ``src/optlib/`` (``constants.py``, ``parsing.py`` [incl. ``parse_args(doc)``], ``params.py``, ``paths.py``, ``optim.py``, ``wandb_utils.py``). The driver adds ``src/`` to ``sys.path`` and imports them; it still owns ``main``, ``run_trial``, ``build_loss_fn``/``build_phase_fns``, ``apply_noise_to_gt``. ``losses.py``/``trial.py`` extraction is a deferred follow-up.

Track-count scaling: ``--batch-size`` (tracks held on GPU per vmap call) and ``--effective-batch-size`` (gradient-accumulation microbatches, cycled via modular indexing over ``track_specs`` in ``run_trial()``) are independent of ``--N-random-tracks`` — there's no vmap over all N tracks at once, just a loop that accumulates. Per wandb runs ``ctyg4znw`` (15 tracks, batch=1, eff_batch=15) and ``hra9nlws`` (100 tracks, batch=1, eff_batch=20), peak GPU memory stayed ~flat (~6.1-6.2 GB) despite the 6.7x track-count increase — memory is governed by ``batch_size``/``effective_batch_size``, not track count, so scaling to e.g. 1000 tracks is memory-safe. Wall-clock time does scale with ``effective_batch_size`` though (~1-1.25 s/step in those two runs); ``hra9nlws`` stopped at step 8250/10000 right around the default Slurm ``--time 03:00:00`` limit, so long-eff-batch sweeps need an explicit longer ``--time`` when submitting via ``submit_jobs_E_field_calibration.py``. ``generate_random_nice_tracks`` (``tools/random_boundary_tracks.py``) now takes ``ke_min_mev``/``ke_max_mev`` (default 100/1000 MeV, unchanged) exposed as ``--track-energy-range-mev MIN MAX`` on ``run_optimization.py``, and as the ``ke_range_mev=(lo, hi)`` kwarg on ``make_efield_calib_command`` in ``submit_jobs_E_field_calibration.py`` (adds a ``Trange{lo}-{hi}`` wandb tag when set).

GT-signal computation (the per-track ``generate_muon_track`` → ``build_deposit_data`` → ``gt_sim.forward`` loop in ``run_optimization.py``, one un-batched call per track) calls ``simulator.forward()``, which runs the **differentiable** path (``_forward_diff``, only ``@jax.remat``-wrapped, not ``@jax.jit``) — so before this fix it ran eagerly (no fused XLA program), paying full per-op dispatch overhead on every track. ``run_optimization.py`` now wraps it once as ``_jitted_gt_forward = jax.jit(gt_sim.forward)`` before the loop. Note ``.forward()``'s static loop bound is the constructor's fixed ``n_segments`` (e.g. ``--gt-max-deposits``), not the per-track ``n_actual`` deposit count, so this compiles once regardless of track length — no per-track recompilation. Clean (pre-noise) GT signals can now be cached to h5 via ``src/optlib/gt_signals.py`` (``save_gt_cache_h5``/``load_gt_cache_h5``, keyed by track name) and ``run_optimization.py``'s ``--gt-cache-save``/``--gt-cache-load``/``--track-shard``/``--exit-after-gt-cache`` flags (relative cache paths resolve under ``$RESULTS_DIR``, matching ``--results-base``'s convention). ``submit_jobs_E_field_calibration.py``'s ``1k_tracks_gt_precompute`` profile precomputes the 2 track ensembles used by ``1k_tracks_sweep`` (rotor weight doesn't affect the GT ensemble), sharded via ``--n-gt-shards`` (default 50 → 20 tracks/shard) and submitted as one dependency chain (``--no-chain`` for independent/parallel); ``1k_tracks_sweep`` then loads that cache by default (``--no-gt-cache`` to recompute; ``--n-gt-shards`` must match between the two invocations). Per-shard host memory scales with tracks-per-shard (each track's GT signal arrays accumulate in Python lists for the whole shard before the h5 flush) — the original 3-shard split (334 tracks/shard) OOM-killed jobs, hence the default was raised to 50 shards. ``profile_100_tracks_sweep`` reuses `_submit_efield_tracks_sweep` (shared with ``profile_1k_tracks_sweep``) to train on only the first 100 of the 1000 tracks: it always requests the full ``--N-random-tracks 1000`` (same deterministic generation as the precompute) then applies ``--track-shard 0 100``, so it reuses the *same* GT cache shards (just the first 5 of 50) with zero new precompute work — no separate 100-track precompute profile needed. ``_n_leading_shards_covering()`` enforces that ``n_active_tracks`` lands exactly on a shard boundary, raising a clear error otherwise (e.g. if ``--n-gt-shards`` is changed to a value that doesn't evenly divide the active-track count).

**Host-memory (not GPU) OOM at N_random_tracks~1000, distinct from the GT-precompute OOM above**: even with ``--gt-cache-load`` (skipping recomputation), ``run_optimization.py``'s GT loop built THREE full per-track lists that live in host RAM for the whole run — ``gt_signals_clean_per_track``, ``gt_signals_per_track`` (noisy), ``gt_weights_per_track``. Each track's full-res signal set is ~116 MB (6 planes, up to 1969 wires x 2701 time bins, float32 — from ``create_sim_config(generate_detector(CONFIG_PATH), total_pad=5000)``), so 3 lists x 1000 tracks ~= 349 GB. Fixed two redundancies (no behavior change, verified via a local smoke run — loss still decreases normally): (1) ``gt_signals_clean_per_track`` is only read by ``--gt-cache-save`` or ``--rotate-noise-seeds > 0``, so it's no longer retained otherwise (``_retain_clean_gt`` flag) — saves ~1/3; (2) Sobolev weights (``make_sobolev_weight``) depend only on ``(shape, max_pad, s, freq_cutoff)``, identical across all tracks unless ``--fourier-cutoff`` masks per-track content, so they're now cached by shape (``_wts_cache_by_shape``) and the SAME array objects are reused across all 1000 slots instead of 1000 independent copies — saves another ~1/3. Post-fix, ``1k_tracks_sweep``/``100_tracks_sweep`` training jobs (neither sets ``--rotate-noise-seeds`` or ``--fourier-cutoff``) needed ~116 GB (just the noisy-signal list) instead of ~349 GB — still large, superseded by the lazy-cache redesign below, which removes the O(N_tracks) host-memory requirement entirely for the ``--gt-cache-load`` path.

**Lazy, disk-backed GT cache (removes the O(N_tracks) host-memory requirement above)**: ``--gt-cache-save``/``--gt-cache-load`` now bake noise into the cached signal (``save_gt_cache_h5(..., gt_signals_per_track, noise_seed=noise_seed)`` — the noisy training target, not the clean pre-noise signal as before), with ``noise_seed`` stored as an h5 file attribute and validated on load (raises if a consumer's derived ``noise_seed`` — from ``SeedSequence(args.seed)``, i.e. tied to ``nn_seed`` — doesn't match what the cache was written with; all current sweep profiles use ``nn_seed=0`` for every job sharing a cache, so this is always satisfied today, but would need a matching cache regen if that ever changes). Old-format caches (no ``noise_seed`` attr) are rejected with a clear error rather than silently mishandled — **this obsoleted all previously-generated ``1k_tracks_gt_precompute`` shards; they must be regenerated** (`submit_jobs_E_field_calibration.py 1k_tracks_gt_precompute --submit`, no ``--only-missing``, since the stale files still exist on disk and would otherwise be skipped).

On the read side, ``run_optimization.py`` (``_using_lazy_gt_cache`` branch) skips the whole per-track generate/forward/noise loop entirely when ``--gt-cache-load`` is set — it only validates cache coverage/seed and derives Sobolev weights from simulator geometry directly (``optlib.gt_signals.LazyGtCache``'s ``__getitem__``, not a real sample array; note ``simulator.forward()`` pads every plane to ``max(num_wires)`` within its volume before stacking, so the weight shape must use that per-volume max, not each plane's true — possibly smaller — wire count, e.g. the Y-plane's 1443 vs U/V's 1969). The "Building deposits" phase-loop then stores each batch's *track name* (not array) in ``build_phase_fns``'s ``gt_lookup`` mode — the returned closure resolves names to arrays fresh on every call via ``LazyGtCache.__getitem__``, so no batch ever retains a real signal array outside the instant it's used. ``LazyGtCache`` bulk-reads one whole shard into memory on first touch. It used to cap at ``max_open_shards=2`` (LRU eviction) on the assumption that batch cycling (``batch_idx = (step*eff_bs+micro) % len(batch_fns)``, window width = ``--effective-batch-size`` batches) stays within ~2 shards' worth of batches — **that assumption is false whenever ``--effective-batch-size`` isn't a clean multiple of a shard's batch count** (``shard_tracks / --batch-size``): the touched-shard set then drifts across the whole shard range over the course of the run (not just within one step), so any fixed cap below the true shard count can still eventually evict and re-read a shard (whole-shard reads, e.g. 2.1GB each for the ke500-1500 20-track shards). Confirmed on wandb run ``6c7vx5s6`` (100_tracks_sweep, ``--batch-size 2 --effective-batch-size 16`` against 5 shards of 20 tracks = 10 batches/shard, i.e. 32 tracks touched per step vs 20 tracks/shard — guaranteed cross-shard every step): real wall-clock (``_runtime`` delta) came out to ~31.7s/step vs a logged ``step_time_s`` of only ~21s, because the shard read happens inside the timed ``fn(p)`` call but ``step_time_s`` is only logged for 1 of every ``log_interval`` steps — a reload landing on any other unlogged step never shows up in the metric, only in total run time. ``run_optimization.py`` now sets ``max_open_shards = len(args.gt_cache_load)`` unconditionally — every shard passed to ``--gt-cache-load`` stays resident for the whole run, no eviction, no re-reads, period (simpler and stronger than trying to size a cap to the batch-cycling window). Memory cost is ``len(args.gt_cache_load) * shard size`` — ~10.5GB for ``100_tracks_sweep``'s 5 shards (fine at the default ``--mem-gb 64``), but ~105GB for ``1k_tracks_sweep``'s full 50 shards, which needs a much larger ``--mem-gb`` when submitting that profile or it'll get OOM-killed by the Slurm cgroup. ``submit_jobs_E_field_calibration.py``'s ``_submit_efield_tracks_sweep`` default ``effective_batch_size`` was also dropped from 16 to 8 (shared by both ``1k_tracks_sweep``/``100_tracks_sweep``) purely to shrink the per-step accumulation window — no longer strictly required for correctness now that all shards stay resident, but keeps per-step compute cost down. Incompatible with ``--rotate-noise-seeds`` (needs multiple realizations; the cache bakes in one) and ``--fourier-cutoff`` (needs each track's own signal for masking) — both raise a clear error if combined with ``--gt-cache-load``.

``tools/eval_efield_track_wireplanes.py`` re-simulates selected tracks (``--track-idx``, default ``0,1,2,3,4,5``) through GT vs the learned SIREN MLP at every saved snapshot (``result['trials'][i]['mlp_trajectory']``, or the single ``live_checkpoint``/``final_p`` fallback — same schema `eval_efield_mlp.py` already reads), saving raw per-(volume,plane) wireplane arrays to one PKL per track (`{stem}_track{idx}_wireplanes.pkl`) for a later HTML viewer. Building the "diff" (learned) ``DetectorSimulator`` requires seeding ``sim._default_sim_params`` with a zero-shaped ``sce_models`` pytree (via ``init_siren``) *before* calling ``warm_up()`` — otherwise the SIREN ``sce_factory`` crashes on ``{**None, ...}`` since warm-up compiles using ``_default_sim_params`` directly (mirrors ``run_optimization.py``'s ``_get_sim``, easy to miss when rebuilding a simulator standalone). Exact sim hyperparameters (``--gt-max-deposits``, ``--max-num-deposits``, ``--num-buckets``, ``--gt-step-size``, ``--step-size``) are parsed back out of the stored ``result['command']`` argv string rather than hardcoded, so the eval matches whatever a given run actually used. Needs a real GPU (full detector-physics forward pass per snapshot) — self-submits via ``--submit``/``--print-commands`` (reusing ``job_submission_tools.s3df_submit``, default partition ``ampere`` = A100, ``gpus=1``) rather than requiring a separate submission script.

``tools/plot_wireplane_eval.py`` builds a lazy-loading HTML viewer from ``*_wireplanes.pkl`` files: run+track / step dropdowns, a 3-row Plotly heatmap grid per plane (learned / GT / learned-GT diff, axes linked via ``matches`` so zooming one syncs the column) cropped to a padded bbox computed from the *learned* arrays only (GT gets real detector noise added everywhere via ``apply_noise_to_gt`` — same draw training compared against — so it's ~100% non-zero and useless for cropping/color-ranging). Also renders 3 downloadable per-row GIFs (500ms/step, matplotlib + Pillow, no ``ffmpeg``/MP4 needed) with download links that update per selected run.

**Deposit-count overflow for the 500-1500 MeV track range** (``RuntimeError: Volume N has X deposits > total_pad``): ``generate_muon_track`` is called without ``detector_bounds_mm`` in ``run_optimization.py``, so at ``--gt-step-size``/``--step-size`` 1.0mm the deposit count is purely energy-determined (dE/dx integration down to ``min_energy_mev=10``) — an exact, direction/geometry-independent ceiling, checkable locally with no GPU via ``generate_muon_track(kinetic_energy_mev=<KE_HI>, step_size_mm=1.0, detector_bounds_mm=None)`` and counting steps. Verified: KE=1000 MeV -> 4409 steps (fits the 5000 default, ~13% headroom); KE=1500 MeV -> 6543 steps (overflows 5000). ``submit_jobs_E_field_calibration.py``'s ``_max_deposits_for_ke(ke_hi)`` now returns 7500 for ``ke_hi > 1000`` (else 5000), threaded through ``make_efield_calib_command``'s new ``max_deposits`` kwarg (sets both ``--gt-max-deposits`` and ``--max-num-deposits``) in both ``profile_1k_tracks_gt_precompute`` and ``_submit_efield_tracks_sweep`` — unaffected profiles (``baseline``, etc., no ``ke_range_mev``) still default to 5000. Existing/completed h5 cache shards for ke500-1500 remain valid regardless (padding only affects computation capacity, not the cached output values; a job either fully completes a shard or crashes with zero partial output) — only the *missing* shards need `1k_tracks_gt_precompute --only-missing` re-run to pick up the fix.

**`6c7vx5s6` (100_tracks_sweep, dep7k) ran ~12x slower per step than `k2wa3z1k` (dep5k) — two compounding causes, not fully disentangled yet.** (1) `--gt-max-deposits`/`--max-num-deposits` 5000→7500 (needed for KE>1000 MeV, per the overflow fix above) increases the differentiable simulator's per-track forward/backward cost — `n_segments`/`response_chunk_size` are forced equal (``tools/simulation.py:118-120``), so `compute_plane_signal` (``tools/physics.py:295-320``) runs the whole deposit budget through one unchunked `fori_loop`. (2) The GT h5 shard cache *was* also thrashing on this run (see the ``LazyGtCache`` note above — every step touched 32 tracks against 20-track shards, guaranteed cross-shard), and since the shard read happens inside the timed ``fn(p)`` call, an initial read of GPU util staying 81-97% with flat GPU memory doesn't actually rule the cache out — a synchronous host-side h5 read still shows up as "the GPU is busy" in a coarse utilization sample if it's interleaved with compute across microbatches. Whether the ~1.5x deposit bump or the shard re-reads dominates the observed 12x is **not disentangled** — both fixes (the `max_open_shards` sizing above, and the deposits change being load-bearing for KE>1000 MeV) are now in place; the new per-shard-read and per-10-step timing prints (`gt_signals.py`'s `_load_shard`, `run_optimization.py`'s step loop) should make this directly visible on the next run instead of requiring after-the-fact wandb archaeology.
